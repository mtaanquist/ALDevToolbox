# Handoff: AL Dev Toolbox — Translator & Reimagined Screens

## Overview

This package contains two related design deliverables for the **AL Dev Toolbox** web app (a tooling suite for AL / Business Central extension development):

1. **Translator** — a production-intent feature for editing XLIFF (`.xlf`, XLIFF v1.2) translation files generated by AL/BC PTE extensions. A translator-grade editor: a list of trans-units, a focused per-unit editor, translation-memory suggestions, filtering, progress visualization, and a keyboard-driven workflow. **This is the primary deliverable.**

2. **Reimagined Screens** — a design-language *study* applying the Translator's refined visual system to two existing screens (**New workspace** and **Object Explorer**). Lower commitment than the Translator; treat as visual direction + interaction reference, not a locked spec.

Both run inside the same app chrome (left sidebar + topbar) and share one design-token stylesheet, so they read as one product.

---

## About the Design Files

**The files in this bundle are design references created in HTML/CSS/JSX — prototypes that demonstrate intended look, layout, and behavior. They are NOT production code to copy directly.**

Specifically, these prototypes:
- Use **in-browser Babel** (`<script type="text/babel">`) to transpile JSX at runtime — fine for design iteration, never for production.
- Share components through **`window.*` global assignments** instead of real module imports (a constraint of the multi-file Babel setup), e.g. `Object.assign(window, { Sidebar, StateBadge, ... })`.
- Pin React via CDN `<script>` tags.
- Use **mock data** hardcoded in `app/data.jsx` and `app/screen-explorer.jsx`.

**Your task:** recreate these designs in the target codebase's existing environment (React, Vue, Svelte, etc.) using its established component patterns, state management, and conventions. If no frontend environment exists yet, pick the most appropriate framework and implement there. Port the *design and behavior* — do not preserve the `window` globals, the Babel script tags, or the CDN React. Convert the CSS custom properties into the codebase's theming system (CSS modules, Tailwind config, styled-components theme, design-token package — whatever is already in use).

## Fidelity

**High-fidelity (hifi).** Final colors, typography, spacing, border-radii, shadows, and interactions are all intentional and specified below. Recreate the UI to match, using the codebase's existing primitives where they exist (buttons, inputs, menus). The exact token values are in the **Design Tokens** section and in `styles/app.css`.

# DELIVERABLE 1 — Translator

**Entry file:** `Translator.html` → loads `app/data.jsx`, `app/icons.jsx`, `app/components.jsx`, `app/layouts.jsx`, `app/app.jsx` + `styles/app.css`, `styles/translator.css`.

## Purpose
Edit a single XLIFF translation file: walk every trans-unit, write/confirm the target-language string, accept translation-memory suggestions, and track progress toward "all translated." Files can be large (~5,000 units) so it's built for keyboard-driven power use.

## Layout
Inside the app content area, top to bottom:

1. **Page header** (`.tr-head`, `--surface`, 22px/40px padding, bottom border)
   - Title row: `Translator` h1 (25px/700) with a `--blue` languages icon; an inline **editable file-name chip** (mono, `--surface-sunken` bg, pencil affordance — click to rename).
   - Sub-row: source language chip `EN · English (en-US)` → arrow → **target-language picker** (see below) · `<file original="…">` in mono · trans-unit count.
   - Right actions: **"Pre-translate from memory"** (secondary, lightning icon) and **"Export .xlf"** (primary blue, download icon).
   - **Progress bar** (`.pviz-bar`): a single segmented bar — final (green) / translated (blue) / review (amber) / to-do (faded amber) — with count chips below and a bold "NN% done" at right.

2. **Filter toolbar** (`.tr-toolbar`, `--surface-2`, bottom border)
   - **Segmented filter tabs**: `Needs translation [n]` / `All [n]` / `Translated [n]` — each with a live count badge. Default = Needs translation.
   - **Kind dropdown** (`All kinds` / Caption / Tooltip / Label / Option) — custom popover, not native `<select>`.
   - **Search** input (icon-prefixed, clear-✕ when populated) — matches source OR target.
   - Right side: "**N hidden by filter**" note, then a **List / Grid view toggle** (segmented).

3. **Body** — one of two views (see below), or an **empty state** (`.tr-empty`) when filters match nothing (search icon, "No units match", "Clear filters" button that resets filter+kind+search).

### View A — "List" (List + detail rail) — DEFAULT, recommended
Two columns: a scrollable **unit list** (`flex: 1`) + a **fixed 408px detail rail** (`--surface`, left border).

- **Unit row** (`.urow`): grid `26px 1fr`, 13px/40px padding, 3px transparent left border.
  - Left: a status dot (9px, colored per state; `fuzzy` gets a ring).
  - Body: meta line (kind mono-tag + unit id in mono `--ink-4`), source text (14px/500 `--ink`), target text (13px `--ink-3`) OR an italic amber "Needs translation" placeholder.
  - **Hover:** bg `--surface-2`. **Selected:** bg `--blue-50`, left border `--blue`.
- **Detail rail** (`.detail`): for the selected unit —
  - Header: "N of M · unit-id" + prev/next nav buttons.
  - **Source block**: `Source · English` micro-label + kind tag; source text in a sunken box; optional developer **note** in an italic left-bordered block.
  - **Target block**: `Target · <language>` micro-label + a state badge; a **textarea** with a blue focus ring; below it a **state picker** (Needs translation / Translated / Final pill toggles).
  - **Suggestions block**: `Suggestions` label + a "Translation memory" pill; a list of suggestion cards — each is numbered (1–9), shows the proposed text, an origin (`Base Application · Microsoft.Sales`, or another extension), and a **similarity badge** (`Exact` green ≥99, `NN%` blue ≥80, `NN%` amber otherwise). Click fills the target + marks Translated. Hover: blue border + ring.
  - **Keyboard legend** (`.kbd-legend`, dashed): documents the chords.

### View B — "Grid" (inline editing, fast pass)
A single table (`.cgrid`): columns = status dot · Unit (id + state badge) · Source · **editable Target cell** (textarea, 38% width). Clicking/focusing a row makes it active (blue bg + the target cell gets the blue focus ring); the active row **expands** a sub-row beneath it showing the developer note + memory suggestions as compact clickable chips (`Memory  [1] text  Exact  origin`). Good for blasting through short captions.

## Target-language picker (`LangSelect`)
The target language in the header is a custom dropdown (used for setting language on a blank new file). Button shows flag tag + "Dansk (da-DK)" + caret; menu lists 12 common BC locales (`LANGS` in `app/data.jsx`: da-DK, de-DE, sv-SE, nb-NO, nl-NL, fr-FR, fr-CA, es-ES, fi-FI, is-IS, it-IT, en-GB) each with a mono flag tag, name, BCP-47 code, and a check on the current one. Selecting it live-updates the header, the editor "Target · …" label + placeholder, and the Grid column header. Closes on outside-click.

## Interactions & Behavior
- **Filtering**: tab (state) AND kind AND search are combined; selection auto-repairs to stay within the filtered set.
- **Fill from suggestion** (click or `Alt+1…9`): sets target text + state→`trans`, fires a toast "Filled from translation memory · marked translated".
- **Pre-translate from memory**: bulk-fills every untranslated/fuzzy unit that has an **Exact (≥99%)** top suggestion, marks them `trans`, toasts the count. (In production this should route fuzzy/non-exact fills through a review step — see Open Questions.)
- **Save & next** (`Alt+Enter`): marks current unit `trans` if target non-empty, jumps to the next unit still needing translation; toast.
- **Rename file**: click the file-name chip → inline input (Enter/blur commits).
- **Keyboard chords** (global, `Alt` + physical `code`): `Alt+↑/↓` prev/next unit · `Alt+Enter` save & next · `Alt+1..9` apply suggestion N.
- **Toast** (`.toast`): bottom-center, dark pill, check icon, auto-dismiss ~1.9s.

## State (current prototype)
`units` (array; each: `id, short, kind, src, note, tgt, state, suggs[]` where a suggestion is `{text, origin, sim}`), `filter` (`todo|all|done`), `kind`, `view` (`list|grid`), `search`, `theme`, `selId`, `fileName`, `tgtLang`, `editingName`, `toast`. In production these split into server data (the parsed XLIFF + TM matches) and local UI state.

## Data / Backend notes (from the original spec)
- Source format is **XLIFF v1.2** (`.xlf`). Each `<trans-unit>` has source, optional target, a `state`, and notes (the developer/context note shown in the editor).
- **Translation memory** suggestions come from base apps (Microsoft.*) **and** the user's own extensions/prior translations, with a similarity score. The `origin` string and `sim` number drive the suggestion UI.
- Export round-trips back to a valid `.xlf` preserving structure.
- **Phase 3 machine-translation assist (BYOK) shipped.** Per-tenant third-party machine translation behind `IMachineTranslationProvider` + `MachineTranslationProviderFactory` (modelled on the off-site storage provider pattern), DeepL first. An org admin configures it under **Administration → Translation**: an encrypted API key (Data Protection key ring) and an `MtTrigger` mode — `Off` (default), `OnDemand`, `AutoWhenNoExactMatch`, or `AlwaysAuto`. Results surface as suggestion chips with an **MT** badge, ranked below memory/in-file matches, and never bulk-applied by "Pre-translate from memory". The AL developer note / kind is passed as the provider's context hint. Exposed to agents via the `machine_translate` MCP tool. There's no universal MT wire protocol, so swappability lives in the provider interface, not an adopted standard.
